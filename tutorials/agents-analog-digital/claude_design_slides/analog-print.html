Why analog sizing is not a grid search

  • Tiny nets. A typical block is 10–30 transistors — far fewer degrees of freedom than a digital flow, but each one matters.
  • Continuous parameters. W, L, bias, mirror ratios. Snap-to-grid is still a continuum.
  • PVT corners. Fast/slow process, ±10% voltage, temperature sweeps. Nominal is a sanity check, not a sign-off.
  • Topology choice is architectural. Miller vs folded cascode vs telescopic is a design decision, not a knob.
Consequence
You cannot random-search this. Every evaluation must start from a good point. That is what gm/ID is for, and what the analog-sizing-advisor does with it.

Part 1 · problem6 / 42

gm/ID — the one-slide methodology

Precompute, per technology + per device, a lookup table of gm/ID (transconductance efficiency) vs VGS − Vth (overdrive).

  • High gm/ID → weak inversion, large W, low noise, area cost.
  • Low gm/ID → strong inversion, small W, fast, power cost.
  • Pick per device; read back W/L and ID from the table.
  • Collapses the sizing problem to one 1-D slider per device.

What an analog agent loop needs

01 Propose
Pick a topology + a sizing vector. Gets better with context.
02 Simulate
ngspice at nominal. Corners come later, post-convergence.
03 Score
Weighted FoM: gain, GBW, PM, power, area.
04 Log
program.md + results.tsv. Human-readable and resumable.
05 Iterate
Propose again conditioned on what we just learned.
Plumbing, not magic
Everything else in eda-agents on the analog side — skills, MCP tools, topology classes, LUT readers — is infrastructure for these five steps.

Cost asymmetry favours greedy refinement

Per SPICE evaluation
~ seconds
ngspice on 10–30 transistors. AC + DC analysis. Measurements parsed inline.
Per LLM proposal
~ cents
Gemini Flash or GLM-4.5-Flash at typical context. Full run budget well under $1.
Consequence for search
You can afford 20–50 SPICE evaluations per run. You cannot afford random exploration. Every proposal must be informed by the current best and the learned insights from prior evaluations — which is exactly what greedy refinement from a gm/ID seed gives you.

analog-sizing-advisor

Analog agent · gm/ID-driven
  • Input: chosen topology + target specs + gm/ID LUTs.
  • Output: starting sizing vector that satisfies first-order constraints before any SPICE call.

Advisor output — first-order sized, no SPICE yet

# Starting sizing vector for a Miller OTA M1_M2: # input pair W: 18e-6 L: 0.5e-6 gm_ID: 15 # low noise M3_M4: # active load W: 4e-6 L: 0.5e-6 M5: # tail W: 8e-6 L: 1.0e-6 gm_ID: 10 # headroom M6: # output stage W: 40e-6 L: 0.5e-6 gm_ID: 8 # drive strength ibias: 40e-6 Cc: 2.0e-12
What just happened
  • gm/ID per transistor was picked from the spec (noise → high, headroom → mid, drive → low).
  • LUT returned W/L and ID for each.
  • Miller cap sized for target PM = 60° using the textbook formula.
  • Ready to hand to the autoresearch runner — no wasted SPICE on random starting points.
First SPICE call confirms
Measures Adc, GBW, PM, power at nominal. If these are within ~20% of the starter specs, greedy refinement takes over. If not, the advisor revises (usually a gm/ID +/- 2 step).

The CircuitTopology contract

class CircuitTopology(ABC): # what the search space looks like def design_space(self) -> dict[str, Parameter]: ... def default_params(self) -> dict[str, Any]: ... # params -> concrete sizing -> ngspice deck def params_to_sizing(self, params) -> Sizing: ... def generate_netlist(self, sizing, work_dir) -> Path: ... # SPICE result -> score + validity def compute_fom(self, spice_result, sizing) -> float: ... def check_validity(self, spice_result) -> list[str]: ... # metadata the harness reads def topology_name(self) -> str: ... def specs_description(self) -> str: ...

Part 3 · architecture18 / 42

The harness derives everything from metadata

  • Prompts: built at runtime from specs_description() + design_space(). No hardcoded names.
  • Tool specs: the MCP describe_topology / evaluate_topology tools dispatch on a class lookup.
  • Validity logic: check_validity() is the single source of truth for "did this sizing meet spec".
  • FoM: each topology owns its weighting — one class, one FoM, no global constants.
What this buys you
Add a new topology tomorrow — every existing agent (recommender, sizing advisor, autoresearch) works against it with zero glue code. The harness is topology-agnostic by construction, not by promise.

analog.corner_validator — after the loop converges

Default corner sweep
AxisPointsNotes
Processtt, ff, ss, fs, sffive SPICE decks per candidate
Voltage−10%, nom, +10%three VDD points
Temperature−40°C, 27°C, 85°Cindustrial range by default
LoadCL/2, CL, 2·CLverifies PM margin under real loads
On failure
If the winner fails any corner, the validator does not accept it. The runner backs off to the previous candidate (row n − 1 in results.tsv) and re-validates. You never ship a nominal-only-passing design.

The full chain

  2. Send a natural-language spec to analog-topology-recommender.
  3. Feed its JSON into analog-sizing-advisor.
