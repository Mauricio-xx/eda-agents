Two design loops, one pattern

Analog loop
  • Inputs — spec (GBW, PM, noise), topology choice.
  • Evaluator — ngspice + gm/ID LUTs.
  • Knobs — W/L, bias current, mirror ratios.
  • FoM — meets spec? area? power?
Digital loop
  • Inputs — RTL, clock, die budget.
  • Evaluator — LibreLane (Yosys + OpenROAD + KLayout).
  • Knobs — density, clock period, PDN pitch.
  • FoM — timing slack, area, power, DRC/LVS green.
Both loops share the pattern: propose → simulate → score → repeat.
eda-agents implements it twice, with the same autoresearch core.

Cost asymmetry drives the strategy

Analog · cheap evaluation
~seconds / eval
  • Budget: tens of SPICE evaluations per run.
  • LLM cost: cents.
  • Search style: fine-grained greedy refinement from a gm/ID seed.
Digital · expensive evaluation
~minutes / eval
  • Budget: a handful of LibreLane runs.
  • LLM cost: dollars.
  • Search style: discrete knob grid, nearest-value snapping.
Same loop. Different budget ceilings. Pick yours in AutoresearchRunner(budget=...).

Three CLI runtimes, one agent codebase

Backend
backend="adk"
  • Python orchestrates 5 agents via Google ADK runner.
  • Locked-down tool set per agent, deterministic model.
  • Good for: CI-style reproducible runs.
Backend
backend="cc_cli"
  • Hands prompt to claude --print.
  • Anthropic subscription; /agents slash command.
  • Good for: interactive Claude Code sessions.
Backend
backend="opencode"
  • Hands prompt to opencode run --format json.
  • Multi-provider: Gemini, OpenRouter, Z.AI, Ollama, …
  • Good for: no Anthropic key, cheap exploration.
Where each backend plugs in
DigitalAutoresearchRunner accepts all four: adk · cc_cli · litellm · opencode — sweeps LibreLane knobs over each.
ProjectManager (master + 4 specialists) currently accepts adk and cc_cli; OpenCode consumes the same skills but through the curated .opencode/agent/*.md roster.

Two files. Resumable runs. No context bloat.

The narrative brain
program.md
# Goal Size a Miller OTA on IHP SG13G2 for GBW ≥ 10 MHz, PM ≥ 60°. # FoM 0.5·GBW + 0.3·PM - 0.2·power # Current best (eval 07) params: {M1.W: 18µ, M1.L: 0.5µ, ibias: 42µA, Cc: 2.1pF} FoM: 0.71 # Learned insights - Miller cap below 2pF ruins PM. - Tail current above 50µA loses gain faster than GBW improves.
The numeric history
results.tsv
eval GBW_MHz PM_deg P_mW FoM 01 6.2 72.1 0.18 0.41 02 8.7 68.4 0.22 0.55 03 11.4 55.3 0.29 0.47 # PM fail 04 10.1 63.8 0.24 0.62 05 12.6 61.2 0.31 0.68 06 10.8 65.4 0.26 0.69 07 11.2 64.1 0.27 0.71 # best

A crashed run resumes by re-reading these two files. The LLM does not need prompt history — it reads program.md and plans the next move.

MCP + skills registry

The stack exposed over one protocol so any MCP-aware client can reuse it.

Mauricio Montanares · IHPPart 05 · MCP
Part 5 · MCP18 / 22

eda-mcp — the portable seam

Stdio MCP server that exposes the repo's primitives by name:

  • Skills — named prompts rendered from Markdown bundles (analog.gmid_sizing, digital.cocotb_testbench, flow.drc_checker).
  • LUTs — gm/ID tables for every registered PDK/device pair.
  • Runners — SPICE, LibreLane, DRC, LVS — same code the agents call internally.
  • Topology registry — list + metadata of circuit topologies.

Without MCP: every consumer depends on the eda_agents package + knows the module layout.
With MCP: the consumer asks for get_skill("analog.gmid_sizing") by name.

Register the server in your client
# Claude Code: ~/.config/claude-code/mcp.json { "mcpServers": { "eda-agents": { "type": "stdio", "command": "eda-mcp" } } } # OpenCode: opencode.json at repo root # auto-bootstrapped by `eda-init`

Templates: src/eda_agents/templates/{mcp,opencode}.json
